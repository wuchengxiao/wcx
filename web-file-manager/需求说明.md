# 网页文件管理程序需求说明

## 1. 项目概述

开发一个浏览器端网页程序，用于管理“临时文本文件”。程序提供文件创建、修改、删除、下载、内容查询等能力，并在页面中展示所有已创建文件的列表，支持在线查看、在线编辑及多选删除。

此处“文本文件”不局限于 `.txt`，还包括可用文本方式打开和编辑的常见文件类型，例如 `.md`、`.js`、`.css`、`.html`、`.json`、`.xml` 等。对于图片、音视频、压缩包、可执行文件等非文本文件类型，页面应给出“暂不支持”提示。

本项目要求将“文件操作函数”与“页面展示逻辑”分离，文件处理能力需放在独立的 JavaScript 文件中，以便后续维护与复用。

## 2. 建设目标

- 提供一个简单直观的网页界面，用于管理临时文本文件。
- 将文件的核心操作封装为独立的 JS 函数模块。
- 支持文件列表展示、内容编辑、关键字检索和批量删除。
- 支持将单个文件下载到本地。

## 3. 适用范围

本程序适用于浏览器环境中的轻量级文本文件管理场景，例如：

- 临时记录文本内容
- 简单草稿编辑
- 本地浏览器会话中的文本文件演示与管理

## 4. 术语说明

- 临时文本文件：由本程序创建并保存在浏览器运行时数据中的文本对象，不要求真实写入用户磁盘。支持 `.txt`、`.md`、`.js`、`.css`、`.html` 等可文本编辑的文件类型。
- 文件列表：程序中所有已创建临时文本文件的集合。
- 在线编辑：用户直接在网页文本编辑区域中修改文件内容。
- 多选删除：用户勾选多个文件后一次性删除。

## 5. 总体要求

### 5.1 功能拆分要求

程序至少包含以下两部分：

1. 文件处理模块
	- 独立 JS 文件。
	- 对外提供文件相关操作函数。
	- 不直接负责页面渲染。

2. 页面交互模块
	- 负责界面展示、事件绑定、用户交互。
	- 调用文件处理模块提供的函数完成业务功能。

### 5.2 浏览器能力边界与持久化说明

由于普通网页无法随心所欲读写用户的硬盘本地文件系统，本系统将所有文件的维护统一托管在**浏览器本地存储（`localStorage`）**中：

- **持久化方案**：底层采用 HTML5 中的 `localStorage`，将临时文件列表对象串行化为 JSON 字符串进行保存。这就保证了即便关闭浏览器、刷新页面，已创建、导入或编辑的文件也不会丢失，拥有高可用性。
- **文件下载实现**：系统通过创建浏览器特有的 `Blob`（二进制大型对象）和带有 `download` 属性的 `<a>` 标签虚拟点击方式，将文本内容和多扩展名打包转换为本地实体文件，实现真实下载。
- **第三方包依赖（若有）**：为了提供 ZIP 打包的能力，程序中需要在 `index.html` 引入来自 CDN（如 `cdnjs`）的第三方 JSZip 库（`jszip.min.js`），文件模块可以无缝调取该对象生成 `.zip` 归档。

## 6. 功能需求

### 6.1 创建临时文本文件

系统应支持创建临时文本文件，至少包含以下属性：

- 唯一标识 ID
- 文件名
- 文件扩展名 / 文件类型
- 文件内容
- 创建时间
- 最近修改时间

创建时应满足以下要求：

- 用户可输入文件名。
- 文件名应包含合法扩展名，或由系统结合用户输入补全默认扩展名。
- 支持创建可文本打开的文件类型，如 `.txt`、`.md`、`.js`、`.css`、`.html`、`.json`、`.xml` 等。
- 当用户尝试创建明显不属于文本类型的文件时，应提示“暂不支持该文件类型”。
- 若未输入内容，可创建空文件。
- 文件创建成功后，文件列表应立即更新。
- 文件名不能为空。
- 如存在重名策略，需在实现时明确：禁止重名或自动重命名，建议默认禁止重名。

### 6.2 修改文件内容

系统应支持修改已创建文件的内容，并满足以下要求：

- 用户可从文件列表中选择某个文件进行编辑。
- 编辑区域应显示当前文件内容。
- 用户保存后，文件内容应更新。
- 最近修改时间应同步更新。
- 修改完成后，文件列表中的相关信息应同步刷新。

### 6.3 删除文件

系统应支持删除单个文件，并满足以下要求：

- 用户可对单个文件执行删除操作。
- 删除前建议进行确认提示。
- 删除成功后，文件列表应立即移除对应记录。
- 若当前正在查看或编辑的文件被删除，页面应清空查看/编辑区域或切换到默认状态。

### 6.4 下载文件

系统应支持单个文件及多个文件的下载，并满足以下要求：

- 用户可对任意一个单一文件执行下载，下载后的文件格式应保持该文件原有扩展名，内容与当前文件一致，且下载文件名默认使用该文件名称。
- 当用户勾选多个文件时，页面顶部或操作区域应提供“下载选中（ZIP）”或“批量下载”功能。
- 批量下载应将选中的多个文件打包成一个 `.zip` 压缩包进行下载，压缩包默认命名为 `archive.zip` 或根据当前下载时间命名。
- 压缩包内的文件应保留其各自的原文件名和扩展名。

### 6.5 查询指定内容

系统应支持按指定内容搜索文件，并满足以下要求：

- 用户可输入关键字进行查询。
- 查询范围至少支持文件内容搜索。
- 建议同时支持按文件名搜索。
- 查询结果应在文件列表中高亮或筛选展示。
- 当查询条件为空时，应恢复显示全部文件。

### 6.6 文件列表展示

页面应展示所有已创建文件的列表，并至少包含以下信息：

- 勾选框
- 文件名
- 创建时间
- 修改时间
- 操作按钮（查看 / 编辑 / 删除 / 下载）

列表应满足以下要求：

- 创建文件后自动追加到列表。
- 删除文件后自动从列表移除。
- 支持选中状态管理。
- 当文件数量较多时，界面应保持可读性，后续实现可增加滚动区域。

### 6.7 在线查看文件

系统应支持在线查看文件内容，并满足以下要求：

- 用户点击“查看”后，可在页面指定区域查看文件内容。
- 查看模式下内容默认只读。
- 应清晰展示当前查看文件的名称。
- **支持按行段查看**：在查看模式下，页面应提供“开始行”和“结束行”的输入框以及“获取指定行”按钮。用户输入行范围并点击按钮后，应调用文件服务模块提供的 `getFileLines` 函数，仅展示该文件指定范围内的文本内容（若输入行号不合法或超出范围，应给出合适提示或进行自动边界裁剪，点击“显示全部”可恢复完整内容）。

### 6.8 在线编辑文件

系统应支持在线编辑文件内容，并满足以下要求：

- 用户点击“编辑”后，可进入编辑状态。
- 编辑区域应支持多行文本输入。
- 用户可保存修改结果。
- 用户可取消编辑并恢复到保存前状态。
- 当用户在编辑状态下点击“放大编辑”进入沉浸式编辑界面后，顶部工具栏应额外提供“自动保存”开关与“保存”按钮。
- “保存”按钮执行的逻辑应与常规保存修改内容完全一致，即立即将当前编辑内容写回对应文件，并同步更新最近修改时间与界面展示状态。
- 当用户开启“自动保存”后，系统应每 5 分钟自动执行一次保存；关闭自动保存后，应立即停止后续自动保存任务。
- 自动保存执行后，工具栏应显示明确的保存反馈，例如“已自动保存”“保存成功”或最近一次保存时间，用于提示本次保存结果。

### 6.9 多选删除文件

系统应支持多选删除，并满足以下要求：

- 文件列表中每条记录前应提供复选框。
- 支持一次勾选多个文件。
- 页面应提供“批量删除”按钮。
- 批量删除前建议给出确认提示。
- 删除成功后，列表应同步刷新。

### 6.10 拖拽导入本地文件

系统应支持从本地拖入多个文件进行导入，并满足以下要求：

- 页面或指定区域应支持 HTML5 Drag and Drop（拖放）事件。
- 用户可从本地操作系统选中一个或多个文件，拖拽并投放到网页指定区域。
- 导入时需读取拖入的文件名称、扩展名及内容，并检查文件类型：
  - 如果是支持的文本文件类型（如 `.txt`、`.md`、`.js`、`.css`、`.html`、`.json`、`.xml`等），则自动解析内容，将其创建并注册到系统的临时文件列表中。
  - 对于不支持的非文本文件类型，系统应给出“暂不支持该文件类型”的温馨提示，且不合并至列表。
- 拖拽导入的文件注册成功后，应与在系统内手动“新建”的文件拥有完全同等的操作权限和数据结构（支持在线查看、在线编辑、单独下载、多选删除和打包下载等）。
- 导入完成后，文件列表应立即自动刷新。

## 7. 文件处理模块需求

文件处理模块必须放在独立 JS 文件中，建议命名为 `fileService.js` 或类似名称。

该模块至少应提供以下函数：

1. `createFile(name, content)`
	- 创建一个新的临时文本文件。

2. `updateFile(id, content)`
	- 根据文件 ID 更新文件内容。

3. `deleteFile(id)`
	- 删除指定文件。

4. `deleteFiles(ids)`
	- 批量删除多个文件。

5. `downloadFile(id)`
	- 下载指定文件。

6. `downloadFiles(ids)`
	- 批量下载多个指定文件，将其打包为 `.zip` 压缩包下载（在前端实现中，通常需要引用诸如 `JSZip` 库或类似方式以支持 ZIP 压缩打包）。

7. `searchFiles(keyword)`
	- 查询包含指定关键字的文件。

8. `getFileById(id)`
	- 根据 ID 获取单个文件信息。

9. `getAllFiles()`
	- 获取全部文件列表。

10. `getFileLines(id, startLine, endLine)`
	- 获取指定文件第 `startLine` 行到第 `endLine` 行之间的内容（从 1 开始计数，包含边界）。
	- 若行号超出范围或不合法，应进行适当处理或截断。

### 7.1 模块要求

- 函数命名清晰，职责单一。
- 页面层不得直接操作底层数据结构。
- 文件处理模块应统一管理文件数据。
- 查询、删除、修改等操作应返回明确结果，便于页面提示成功或失败。

## 8. 页面需求

页面至少包含以下区域：

### 8.1 顶部操作区

- 新建文件输入项
- 文件初始内容输入项（可选）
- 创建按钮
- 拖拽投放区（拖拽文件放置区域，支持拖入多个外部文本文件）
- 搜索输入框
- 搜索按钮或实时搜索能力
- 批量删除按钮
- 批量下载按钮（支持将选中文件打包为 `.zip` 下载）

### 8.2 文件列表区

- 展示全部文件
- 提供多选框
- 提供查看、编辑、删除、下载操作入口

### 8.3 内容展示 / 编辑区

- 显示当前文件名与文件信息
- 文本内容区域（在线查看/编辑区）
- 查看状态与编辑状态区分明显
- **行过滤控制项（仅在查看模式可用）**：提供“开始行（1-based）”和“结束行”两个数字输入框，以及“获取特定行”和“显示全文”按钮，用于测试并触发 `getFileLines`。
- **编辑区尺寸管理**：新建文件表单中的文本输入区，以及在线查看/编辑工作区，均需配备独立的“放大编辑”与“还原大小”按钮。用户点击放大时，编辑区将以全屏幕沉浸式大卡片形式浮于页面最前端以供精细编辑，还原时应保留已有文本输入，并收回原位。
- **放大编辑工具栏**：进入“放大编辑”后，工具栏至少应包含“自动保存”开关、“保存”按钮、当前保存状态展示区；保存状态展示区应能体现手动保存或自动保存的执行结果。
- 编辑状态下提供保存与取消按钮

## 9. 交互需求

- 所有的常规成功反馈和提示消隐应优先采用**非阻塞式 Toast 提示框**形式。
- 原生弹出点击不应打断用户操作流，对于“成功”反馈（如创建成功、修改成功、下载就绪），信息在界面提示区静默滑入并展示固定时间（如 3-4 秒）后应平滑淡出，无需用户点击确认。
- 对于失败、由于重名产生的异常警示提示，为防用户遗漏，可以延长淡出展示停留时间（如 5 秒左右），并在页面显眼处标识，随后自动消退。
- 当未选择任何文件却点击批量删除或批量下载时，应给出合适消隐提示。
- 当检索关键字没有匹配文卷时，列表应输出友好缺省标识。
- 当用户点击放大编辑工具栏中的“保存”按钮时，应复用现有保存修改逻辑，不得引入与常规保存不一致的数据写入路径。
- 当自动保存开关处于开启状态时，系统应以 5 分钟为周期执行保存，并在每次执行后通过工具栏状态区或 Toast 明确反馈保存成功、失败或最近保存时间。

## 10. 数据要求

每个文件对象建议至少包含以下字段：

- `id`
- `name`
- `extension`
- `mimeType`（可选，根据实现决定是否保存）
- `content`
- `createdAt`
- `updatedAt`

可选扩展字段：

- `selected`
- `size`
- `type`

## 11. 非功能需求

### 11.1 易用性

- 页面布局清晰。
- 按钮命名直观。
- 主要功能应在同一页面完成，无需跳转。

### 11.2 可维护性

- 文件处理逻辑与页面逻辑分离。
- 代码结构清晰，便于后续增加重命名、导入文件、排序等功能。

### 11.3 兼容性

- 支持现代主流桌面浏览器。
- 下载功能应兼容常见浏览器的 Blob 下载方式。

### 11.4 性能

- 在少量至中等数量文本文件场景下，页面操作应响应及时。
- 创建、删除、编辑、搜索后界面应及时刷新。

## 12. 建议的目录结构

后续实现时建议采用如下结构：

- `index.html`：页面结构
- `style.css`：页面样式
- `fileService.js`：文件处理模块
- `app.js`：页面交互逻辑

## 13. 验收标准

满足以下条件可视为需求完成：

- 可以创建临时文本文件。
- 可以创建多种可文本编辑的文件类型，如 `.txt`、`.md`、`.js`、`.css`、`.html`。
- 可以查看文件列表，且列表会实时更新。
- 可以在线查看文件内容。
- 可以在线编辑并保存文件内容。
- 可以删除单个文件。
- 可以多选并批量删除文件。
- 可以下载单个文件到本地。
- 可以多选文件并打包成一个 `.zip` 文件下载。
- 可以从本地拖拽一个或多个文本文件（如 `.md`, `.js` 等）入网页区域并成功解析导入。
- 拖入的文本文件在列表中拥有完全一样的编辑、查看、下载和删除等操作权限。
- 可以按关键字查询文件名或文件内容。
- 支持获取并查看指定文件任意起始行到结束行的部分行内容。
- 新建文件区及右侧编辑区，均配置独立的“放大编辑与还原大小”功能。
- 所有非破坏性成功的交互事件（如新建、修改、单/多下载、拖入成功），均采用非阻塞式的 Toast 浮动通知，并在一段时间后自动渐隐（无需用户手动点击关闭）。
- 对非文本文件类型会提示暂不支持。
- 文件操作逻辑已拆分到独立 JS 文件中。
